How to use Jenkins sharedLibrary

Aug 31, 2018

At early stage if we use Jenkins for deployment, each code repository often needed to contain a Jenkinsfile, which defined a large amount of repetitive pipeline code. However, by extracting this repetitive code, we can form a common module, known as the Jenkins Shared Library.

The term “Jenkins Shared Library” can be confusing at first glance. For easier understanding, it can be simply considered as a public module, similar to a second-party package, to achieve logic reuse in pipelines.

The structure of the Shared Library

In my view, the Shared Library is a specific development framework for jenkins pipeline. Let’s take a look at the structure of the Shared Library as the official defined.

(root)
+- src                     # Groovy source files
|   +- org
|       +- foo
|           +- Bar.groovy  # for org.foo.Bar class
+- vars
|   +- foo.groovy          # for global 'foo' variable
|   +- foo.txt             # help for 'foo' variable
+- resources               # resource files (external libraries only)
|   +- org
|       +- foo
|           +- bar.json    # static helper data for org.foo.Bar
  • var place pubic functions or logic, generally used to define the main implementation code of the pipeline.
  • resources place resource files, serving the same purpose as the ‘resources’ directory in a Java project.
  • src place utility functions commonly used for tasks like alerts and message notifications.

Merely understanding it conceptually can be confusing. Let’s illustrate how to use a Shared Library with some examples.

The “degradation” of the Jenkinsfile

To support application deployment, a common practice in the early days was to create a Jenkinsfile at the project’s root directory and define the pipeline implementation logic within it.

With the introduction of Shared Library functionality, it’s logical to move the reusable logic from the Jenkinsfile to global variables in a ‘vars’ directory. The original Jenkinsfile then only needs to define the deployment parameters.

As shown below, we define a global variable named ‘javaDeploy.groovy’ in the ‘vars’ directory and override the call() method to implement the deployment logic for the pipeline:

cat var/JavaDeploy.groovy

def call(Object WorkflowScript) {
    // Receive parameters configured in the Jenkinsfile
    def service = WorkflowScript.hasProperty('service') ? WorkflowScript.service : ''
    def port = WorkflowScript.hasProperty('port') ? WorkflowScript.port : ''
    def jvmOptions = WorkflowScript.hasProperty('jvmOptions') ? WorkflowScript.jvmOptions: '-server -Xms512m -Xmx1g -XX:MetaspaceSize=128m -XX:MaxMetaspaceSize=512m'
    
    pipeline {
      agent any
      stages {
        stage('Git Clone') {
            steps {
                checkout scm
            }
        }
        stage('Build') {
            steps {
              ...
            }
        }
        stage('Push image') {
            steps {
              ...
            }
        }
        stage('Deploy'){
            steps {
             ...
            }
        }
      }
    }

The original Jenkinsfile only needs to configure the relevant deployment parameters and call javaDeploy with the parameters at the end:

cat Jenkinsfile

import groovy.transform.Field

// service name 
@Field
String service = 'demo'

// listen port for service 
@Field
String port = '9903'

// The default value for JVM Parameters
@Field
String jvmOptions = '-server -Xms512m -Xmx1g -XX:MetaspaceSize=128m -XX:MaxMetaspaceSize=512m'



// Calling the global variable
javaDeploy(this)

The logic of the Shared Library is also stored in the repository just like business code. When the Jenkins job runs, it loads the code from this repository.

As a result, the content of Jenkinsfiles becomes simpler, essentially turning into configuration files. Developers only need to configure a few parameters, while operations personnel only need to maintain the logic in the Shared Library.

Introduction of the Kubernetes plugin

The company’s original Jenkins system had a master-slave structure with one master and three slaves. However, it struggled to handle the daily release volume averaging hundreds of times, often crashing due to out-of-memory (OOM) issues. Despite several hardware upgrades, the fundamental problem remained unresolved. Therefore, considering cost and performance, we decided to introduce a Kubernetes (K8s) plugin.

By leveraging Kubernetes’ elastic scaling capabilities, we implemented fast horizontal scaling of agents. Whenever there’s a build requirement, an agent pod is created to execute the task. Once the task is completed, the agent pod is immediately destroyed:

For how to use the kubernetes-plugin, the official documentation provides a demo, using Declarative Pipeline as an example:

pipeline {
  agent {
    kubernetes {
      yaml '''
        apiVersion: v1
        kind: Pod
        metadata:
          labels:
            some-label: some-label-value
        spec:
          containers:
          - name: maven
            image: maven:alpine
            command:
            - cat
            tty: true
          - name: busybox
            image: busybox
            command:
            - cat
            tty: true
        '''
    }
  }

You can see there’s a large section defining the agent pod, which is unrelated to the main logic and makes the code cumbersome. For the sake of simplicity, we can extract this pod definition into a separate file and then include it in the main logic of the pipeline.

So, how do we do it?

First, write the defination of the agent pod into a seperate yaml file(jenkinsAgent.yaml) and put it in the resources dir

cat resources/com/sunny/jenkinsAgent.yaml
apiVersion: v1
    kind: Pod
    metadata:
      labels:
        some-label: some-label-value
    spec:
      containers:
      - name: maven
        image: maven:alpine
        command:
        - cat
        tty: true
      - name: busybox
        image: busybox
        command:
        - cat
        tty: true

Then, in the Jenkinsfile, import the Agentyaml by using the libraryResource that method built into the pipeline to load the file and convert it to a string variable:

 @Field
 def jenkinsAgentYaml = libraryResource 'com/sunny/java/jenkinsAgent.yaml'

 pipeline {
     agent {
       // enable kubernetes plugin
       kubernetes{
         yaml jenkinsAgentYaml
       }
     }
   ...
}

By this, the code look more concise and streamlined.

Include Dockerfile

Regarding the Dockerfile, it’s conventional to place it in the same repository as the business code. However, considering that most projects in the company use similar Dockerfiles, we’ve also extracted this common logic and saved it in the Jenkins Shared Library. It’s stored in the resources directory and called and processed in the main logic of the pipeline.

Here’s the Dockerfile for a Java project:

FROM BASE_IMAGE
MAINTAINER beiguo <beiguo@heaven.com>

ARG module_name
ADD ${module_name}.jar /
ADD entrypoint.sh /
ENV module=${module_name}
ENTRYPOINT ["sh","/entrypoint.sh"]

The startup script entrypoint.sh used in it is also placed in the resources directory.

Then, call it in the Jenkinsfile:

// load the Dockerfile
@Field 
def dockerFile = libraryResource 'com/sunny/java/Dockerfile'

pipeline {
      ...
        stage('Build') {
            steps {
                    ...
                    writeFile file: "Dockerfile",text: dockerFile
                    sh "docker build --build-arg module_name=${params.module_name} -t sunny.com/java/demo"
                    ...
                }
            }
        }

Dynamically Generating JVM Parameters

Different environments like production, testing, and staging use varying JVM parameters. Therefore, we need to dynamically generate JVM parameters in the pipeline as well. We can use Groovy to define a method to achieve this. Such utility functions are recommended to be placed in the src directory.

src/com/sunny/Utils.groovy

package com.sunny

class Utils implements Serializable{

     /**
     * generate jvm parameters based on the deployment environment.
     * @param deployEnv 
     * @param jvmOpts jvm堆栈空间参数 p.s. -server -Xms512m -Xmx512m 
     * @return
     */
    def generateJavaOpts(String deployEnv,String jvmOpts,String naocsConfigServer) {
        
        def logDir="/log/${module}"
        def sentinelService="sentinel:8280"
        def skywalkingAgent="/usr/skywalking/agent/skywalking-agent.jar"
        def skywalkingBackend="oap.skywalking:11800"
        def skywalkingServiceName="${module}"

        // JVM GC options
        def gcOpts=("-verbose:gc"
                       +" -XX:+PrintGCDetails"
                       +" -XX:+PrintGCDateStamps"
                       +" -XX:+PrintGCTimeStamps"
                       +" -XX:+UseGCLogFileRotation"
                       +" -XX:NumberOfGCLogFiles=100"
                       +" -XX:GCLogFileSize=100K"
                       +" -Xloggc:${logDir}/gc.log"
                       +" -DLOG_PATH=${logDir}/log.log")
        // HeapDump options
        def dumpOpts="-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=${logDir}"
        // sentinel options
        def sentinelOpts="-Dcsp.sentinel.api.port=8719 -Dcsp.sentinel.dashboard.server=${sentinelService}"
        // nacos options
        def nacosOpts="-DNACOS_CONFIG_SERVER=${naocsConfigServer}"
        // skywalking options
        def skywalkingOpts=("-Dskywalking.agent.service_name=${skywalkingServiceName}"
                          + " -Dskywalking.collector.backend_service=${skywalkingBackend}"
                          + " -javaagent:${skywalkingAgent}")
        def javaOpts = ''
        switch(deployEnv){
            case "test":
               javaOpts = "${jvmOpts} ${gcOpts} ${dumpOpts} ${nacosOpts} -Dproject.name=${projectName}"
               break
            case "production":
               javaOpts = "${jvmOpts} ${gcOpts} ${dumpOpts} ${sentinelOpts} ${nacosOpts} ${skywalkingOpts} -Dproject.name=${projectName}"
               break
            ...
        }
        return javaOpts
    }
}

Conclusion

In addition to the functionalities mentioned above, leveraging Jenkins Shared Library allows us to implement various flexible and versatile requirements, such as dynamically generating Kubernetes YAML manifest files and more.
